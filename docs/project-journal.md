## 2026-07-01 12:00 Asia/Magadan — оценена достаточность PostgreSQL-данных для реальных отчётов по малой выборке

**Статус:** выполнено. Код не изменялся. Ветка не создавалась. Новая версия не выпускалась.

Вопрос пользователя: достаточно ли текущих данных в PostgreSQL для формирования реальных полных отчётов по небольшой выборке приборов.

Ответ:
- для `Instrument Specification` по малой выборке данных достаточно, особенно по прибору `RURN-NC-QAY52174 -FIT 01`;
- для `Process Data` данных достаточно для реального отчёта по имеющимся строкам;
- для простого `I/O Map` данных достаточно для ограниченной реальной выборки;
- для простого `I/O Tag Assignment` данных достаточно для ограниченной реальной выборки;
- для простого `Cable / Wiring Schedule` данных достаточно по части кабелей, пар/жил и вероятных клемм;
- для полноценного `Point-to-Point Wiring Diagram`, SmartLoop, Loop Signal Diagram и Tag Signal Diagram данных пока недостаточно, если требовать инженерно точную графическую схему на уровне SPI.

Важное уточнение:
- можно формировать реальные отчёты по текущей PostgreSQL-выборке;
- нельзя обещать полное 1:1 повторение всех SPI-отчётов, потому что часть полей SPI отсутствует или хранится пока в JSON/демо-структурах;
- правильная цель `v0.8.5`: сделать реальные отчёты по имеющимся данным, с пустыми/прочеркнутыми полями там, где данные ещё не загружены из Oracle SPI.

Что есть в текущей PostgreSQL-модели:
- структура Plant / Area / Unit / Loop / Instrument;
- приборы FIT/PIT/TIT по `RURN-NC-QAY52174`;
- спецификации с JSON-полями;
- process data;
- документы и ревизии;
- панели, шкафы, terminal strips, terminals;
- кабели, cable sets, wires;
- I/O rack / slot / apparatus / channel;
- control system tags;
- source trace через `source_objects` и `source_object_links`.

Что можно делать в `v0.8.5`:
1. Реальный `Instrument Specification` по FIT/PIT/TIT, с лучшей полнотой по FIT.
2. Реальный `Process Data` report по текущим строкам.
3. Простой реальный `I/O Map` по channel/control system tag/rack/slot/card.
4. Простой реальный `I/O Tag Assignment` по tag/channel/terminal strip/terminal.
5. Простой `Cable / Wiring Schedule` по cable/cable_set/wire/probable_terminal.

Что не делать в `v0.8.5`:
- полноценную графическую генерацию SmartLoop;
- полноценный Point-to-Point Wiring Diagram;
- Loop Signal Diagram / Tag Signal Diagram;
- полное повторение SPI title block и всех revision blocks, если данные отсутствуют.

Решение:
- `v0.8.5` можно утверждать и начинать;
- перед началом разработки нужно понимать: отчёты будут полными по текущей малой PostgreSQL-выборке, но с явным отображением отсутствующих полей как пустых/`—`/`Нет данных`;
- Oracle дообследование вести параллельно, но не блокировать `v0.8.5`.

Следующая команда, которую ждём от пользователя:
- `Утверждаю v0.8.5. Создай ветку и начинай реализацию.`

## 2026-07-01 11:45 Asia/Magadan — принято решение по дообследованию Oracle SPI DB

**Статус:** решение зафиксировано.

Решение:
- для старта `v0.8.5` дообследование Oracle SPI DB не является блокером;
- `v0.8.5` можно начинать на текущей PostgreSQL-модели и загруженных эталонных отчётах;

Что нужно дообследовать в Oracle SPI DB параллельно:
1. Где хранятся приборы / tag numbers.
2. Где хранятся loop numbers и связь Instrument → Loop.
3. Где лежат specification forms и значения полей спецификаций.
4. Где лежат process data.
5. Где лежат I/O assignments.
6. Где лежат cable / wiring / terminal data.
7. Как связаны Instrument → Cable → Core → Terminal → Panel → I/O Channel.
8. Как SPI хранит report/document numbers и revision blocks.
9. Как `OBJECT_REGISTRY.OBJECT_ID` / `OBJECT_UID` связаны с engineering tables.

## 2026-07-01 11:30 Asia/Magadan — подготовлен список изменений для `v0.8.5` по отчётам без изменения кода

**Статус:** выполнено. Код не изменялся. Ветка не создавалась. Новая версия не выпускалась.

Цель `v0.8.5`:
- сделать первые реальные отчёты из PostgreSQL на базе уже существующей инженерной модели;
- не пытаться полностью повторить графические SPI-отчёты;
- подготовить демонстрационный результат, который можно показать как движение от карточки прибора к реальным отчётам.

Состав `v0.8.5`:
1. Раздел `Отчёты` в интерфейсе.
2. Отчёт `Instrument Specification`.
3. Расширение правой панели свойств прибора как источника данных для отчёта.
4. Простой табличный `I/O Map`.
5. Простой табличный `I/O Tag Assignment`.
6. Простой `Cable / Wiring Schedule`, если текущих данных достаточно.
7. Backend API для отчётов.
8. Минимальные Alembic-изменения при необходимости.
9. Demo data для отчётов.
10. Frontend просмотр отчётов.
11. Проверки через Docker Desktop.
12. Документация и журнал.

Не входит в `v0.8.5`:
- полноценная генерация SmartLoop;
- Enhanced SmartLoop;
- Loop Signal Diagram;
- Tag Signal Diagram;
- Point-to-Point Wiring Diagram;
- полноценная модель трассировки сигналов;
- полный редактор шаблонов отчётов;
- PDF-экспорт, если это задержит рабочий прототип.

Критерий готовности `v0.8.5`:
- пользователь запускает прототип через Docker Desktop;
- открывает прибор;
- видит SPI-подобные свойства прибора;
- открывает `Instrument Specification` из PostgreSQL;
- открывает простые `I/O Map` и `I/O Tag Assignment`;
- версия видна как `0.8.5`.
